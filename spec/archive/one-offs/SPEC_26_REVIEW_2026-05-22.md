# SPEC 26 — Review & Recommendation (2026-05-22)

> Founder asked: *"I want you to review SPEC_26_MPP_SETTLE_ON_SUCCESS.md — it's already archived but I think we should simplify this and revert this change. Thoughts?"*
>
> Honest answer: **don't revert wholesale, but the dual-path complexity is a real cost and the founder's instinct is half right.** Below: what SPEC 26 bought, what it cost, what's load-bearing, what's not, and what I'd actually change.

---

## 1. What SPEC 26 bought (still real)

The structural win is one sentence: **`bug_mpp_no_refund_on_failure` went from 2 incidents in 24h on a single route to 0 across 68 routes.** That's the receipt that justified the spec, and it still holds.

Concretely:
- 256x256 → 400 from OpenAI, $0.05 charged → user out the money. Pre-SPEC-26.
- Same case post-SPEC-26 → HTTP 402, no charge, LLM retries with corrected size for free. Post-SPEC-26.

Non-trivial when you scale this to Audric Store. Music gen ($0.50), video gen ($5), ebook gen ($1) — at those amounts a single paid failure is a refund ticket, not a "$0.05 oh well." Reverting means accepting that surface back, then patching it in app code per-vendor.

The probe-then-charge ORDER is the load-bearing invariant. Don't revert that.

---

## 2. What SPEC 26 cost (also still real)

The hotfix log is dense. Reading it carefully, three real costs surface:

### 2.1 Double-probe in audric prepare/complete (~2x vendor spend per pay)

This is the single biggest unsolved cost. Today's flow:

```
audric prepare:   probe upstream  →  get verdict  →  return 402 to client
                  ↓ (vendor cost #1)
client signs tx
audric complete:  probe upstream  →  get verdict  →  charge → return body
                  ↓ (vendor cost #2)
```

**Every successful pay flow incurs the upstream vendor cost twice** because the prepare-time probe result is discarded and the complete-time probe re-fires. At $0.05/probe (image gen), that's $0.10 spent for $0.05 revenue. Gateway absorbs the difference.

The hotfix log explicitly acknowledges this and explicitly defers it: *"Defer until vendor-cost telemetry shows it matters."* I'd argue the telemetry doesn't need to "trend bad" — the structural waste is 100%, every flow, no exceptions. Audric Store's $5 video-gen route would absorb $5 of waste per successful $5 sale = -100% margin.

### 2.2 Dual-path maintenance burden

68 chargeProxy routes are on `settleOnSuccess: true`. 21 chargeCustom routes are on the legacy charge-first path. Every gateway change has to consider both paths. Every test needs both branches. Every operator-readable log has to be interpreted under both flows.

The spec says chargeCustom migration is "out of scope for SPEC 26 v1 since the option doesn't exist on `chargeCustom` yet (follow-up)." It's still a follow-up. The dual path is now permanent if nothing forces consolidation.

### 2.3 Code-path complexity bred 4 hotfixes in 24h

- **v1.0.1** — credential-presence guard ATTEMPTED + REVERTED. Founder reasoning seemed sound (skip probes when no auth header); regression introduced a money-loss class. The spec correctly identifies *"the prepare-time probe is the safety gate"* as load-bearing.
- **v1.0.2** — probe-throw safety net + telemetry split + quality validate. Mixed-bag of defensive fixes.
- **v1.0.3** — audric prepare 4xx pass-through. Pre-existing audric bug exposed by SPEC 26 surfacing real errors.
- **v1.0.4** — DALL-E brand scrub. Tangential to SPEC 26.

Of these, only v1.0.1 was a SPEC-26-architectural bug. The other 3 are either defensive maintenance (v1.0.2) or things SPEC 26 surfaced rather than caused (v1.0.3, v1.0.4). So the architectural complexity isn't fatal — but it IS expensive.

## 3. What I'd actually do (recommendation)

Don't revert SPEC 26. Don't keep it at the status quo. Three concrete changes, in priority order:

### 3.1 (P0, BEFORE Audric Store launch) — Fix the double-probe via prepare-time response cache

The hotfix log already named the fix:

> *"The cleanest fix is to make the prepare-time probe write to the response cache so the complete-time call hits the cache → single probe per flow."*

The structural change is small. Add ~30 lines to `chargeProxySettleOnSuccess`:

```typescript
// During probe, cache the response by fingerprint
if (verdict.kind === 'deliverable') {
  await responseCache.set(fingerprint, { body, status, headers }, 60s)
}

// At charge time, hit cache before re-probing
const cached = await responseCache.get(fingerprint)
if (cached) return chargeAndDeliver(cached)
```

Cuts vendor cost in half on every pay flow. Only caches `deliverable` verdicts (refundable/mixed re-probe at charge time, which is right — failure modes might be transient). The 60s TTL is already wired (Upstash Redis). Just plumb the prepare-time write into it.

**Effort: ~½d. Test impact: 5–10 new tests. Vendor cost saving: ~50% of all gateway probes.**

This single change is more valuable than every other SPEC-26 follow-up combined.

### 3.2 (P1, parallel to Audric Store) — Drop dual path: settle-first becomes the only path

Two ways:

**Option A — migrate chargeCustom to settle-first.** ~1d. Adds settleOnSuccess + classifier framework support to chargeCustom. All 89 routes on one model. Maintenance complexity collapses to one branch.

**Option B — drop settleOnSuccess from chargeProxy, going back to charge-first everywhere.** ~½d. Loses every SPEC-26 win. Audric Store risk goes from "$0 paid failures" to "1–5% paid failure rate at $0.50–$5/call." Don't pick this.

**Recommendation: Option A.** The 68 settle-first routes are working. chargeCustom is the holdout. Forcing one model means every operator only has to learn one mental model. The 21 chargeCustom routes are commerce APIs (Lob, Printful) that especially benefit from settle-first because their dollar amounts are higher.

### 3.3 (P2, defer until justified) — Strip partial-success classifier from non-images routes

D-6 added a partial-success classifier framework so OpenAI image gen could fractional-charge for n=4 with 3 successes. **No other production route uses partial-success classification.** The framework is general-purpose machinery sitting on every route's classifier path even though only one route exercises it.

If the framework feels load-bearing in 2 months because more partial-success routes appear, keep it. If it stays unused, strip it back to a per-route bypass for images only. Not urgent. Not blocking. Just code-debt-tracking.

---

## 4. What I would NOT do

- **Revert SPEC 26 wholesale.** The structural win (zero paid failures on synchronous vendor errors) is real and load-bearing for Audric Store. Reverting means accepting the surface back and patching it in app code per-vendor. That's worse architecture, not simpler.
- **Promote `refund(digest)` MPP contract primitive now.** O-4 in the spec defers it correctly. It's a multi-week scope. Async-fulfillment failures (Lob 3-day-later print fail) are not the urgent failure mode — they're rare and acknowledged. Revisit when Audric Store's Lob/Printful traffic > 10/day.
- **Split out telemetry "again."** v1.0.2's `event=charge_succeeded` separation already solved the v1.0.1 "founder counted probes as charges" trap. The current split is good enough. Don't fork the helper a third time.

---

## 5. The TL;DR I'd give the founder

> Your instinct that SPEC 26 is over-engineered is half right. The probe-first ORDER is load-bearing — it's why Audric Store can charge $5/video-gen without a refund ticket every other flow. **Don't revert that.** But the double-probe (prepare + complete both fire upstream) doubles vendor cost on every pay, and the dual chargeProxy/chargeCustom path is permanent maintenance debt. Fix those two, in that order, and SPEC 26 stops feeling complex because it stops doing the wasteful thing.

## 6. If you want a different answer

If you read this and still want to revert: **what's the trigger?** What would have to be true for "revert + patch in app code" to win? My read: only if vendor-failure rate is < 0.5% AND average call value is < $0.20 forever. Audric Store's roadmap explicitly breaks both conditions ($0.50–$5 calls, with hosted commerce APIs that have higher failure rates than OpenAI). So the math says keep settle-first; just stop paying for it twice.

If your real gripe is "the spec doc is too long" — yes, 716 lines + 4 hotfix logs is a lot. That's a documentation cleanup, not an architectural revert. We could fold the hotfix logs into a `lessons-learned` section in 50 lines and archive the rest.
