## Purpose under v0.7a

The v0.6 plan audited 130 behaviors across `audric/apps/web/app/api/engine/chat/route.ts` (1,600 LoC, 62 server behaviors) and `audric/apps/web/hooks/useEngine.ts` (1,924 LoC, 68 client behaviors). Each behavior was assigned a destination in v0.6's full audric/web chat-shell migration.

**v0.7a is a DIFFERENT rollout.** Per the v0.7a plan, `@t2000/engine` internals drain onto Vercel AI SDK while preserving the engine's public API (`QueryEngine`, `submitMessage`, `EngineEvent`, `PendingAction`). `audric/web` does NOT change substantively — the **sole** audric-side modification is `voice/transcribe/route.ts` (R3), which lands as a tiny concurrent PR.

The catalogue's value for v0.7a is therefore narrower but sharper:

1. **Establish the contract surface.** Every v0.7a engine PR must continue to support all 130 audric/web behaviors that today consume the engine's `EngineEvent` stream + SSE wire format. The bridge layer (`packages/engine/src/bridge/`) is the seam that guarantees this.
2. **Identify engine-internal subset.** Of the 130 behaviors, only those that involve the engine's own emit shape, narration markers, attemptId stamping, or tool I/O contract are touched by v0.7a. The rest live in audric/web and are passive consumers.
3. **Per-phase verification matrix.** Each phase release (1–8) names the behaviors it touches; the next agent verifies the bridge preserves them via tests + smoke before merging.
4. **Discovery channel.** New behaviors surfaced during execution get added to this file with a v0.7a interpretation BEFORE implementation.

## Legend (preserved from v0.6 + v0.7a column)

**v0.6 destination codes** (audric/web migration):
- **P** — Preserved as-is in same module
- **F** — Preserved in `useEngine` façade or thin shim
- **M** — Moved to new module
- **R** — Replaced by AI SDK API
- **D** — Deleted (no longer needed)
- **NEW** — New functionality requested by audit

**v0.7a interpretation codes** (engine drain):
- **engine-internal** — Behavior emits or consumes EngineEvent / SSE shape that the engine produces. v0.7a phases 1–5+ may touch the implementation; bridge layer must preserve the wire shape.
- **audric-web** — Lives in audric/web and is untouched by v0.7a. Listed for completeness so PR authors can confirm "no engine impact."
- **R3** — The single audric/web file that DOES change in v0.7a (voice/transcribe).
- **deferred** — Marked D in v0.6 (deleted) or covered by a follow-up SPEC; no v0.7a action.

## v0.7a interpretation — what's actually engine-internal

Of the 130 behaviors enumerated below, the engine-internal subset that v0.7a directly affects is:

**Server-side (engine-internal):**
- S28 — Manual `ReadableStream` + SSE encoder → engine `streaming.ts` produces SSEEvent; v0.7a Phase 5 routes this through `createUIMessageStream` then back through `bridge/sse-format-adapter.ts` for byte-equivalence.
- S40 — `harnessShape` emit → engine `EngineEvent.harness_shape`.
- S42 — Main agent loop emits engine events → entire Phase 1–3 surface (provider migration + tool migration + `streamText` adoption).
- S43 — Event tap for metrics covering the FULL EngineEvent set: `text_delta`, `thinking_delta`/`thinking_done`, `tool_start`, `tool_progress`, `todo_update`, `pending_input`, `tool_result`, `usage`, `turn_complete`, `pending_action`, `harness_shape`, `compaction`, `error` → bridge layer's `event-bridge.ts` is the v0.7a embodiment of this.
- S44 — `attemptId` stamping on `pending_action` → MANDATORY per `agent-harness-spec.mdc` Item 3; bridge does NOT stamp (engine orchestration does).
- S47 — SSE serialization policy (skip compaction, skip deduped tool_result, sanitize errors) → preserved by `bridge/sse-format-adapter.ts` via `streaming.ts:serializeSSE`.

**Client-side (engine-internal — i.e., touches the wire format the engine produces):**
- C38 thinking_delta / C39 thinking_done — engine emits these; bridge preserves `blockIndex` + Anthropic signature.
- C40 text_delta — engine emits; bridge preserves.
- C41 tool_start — engine emits; bridge preserves.
- C42 tool_result — engine emits with dedup metadata; bridge preserves dedup signal via `serializeSSE` filter.
- C43 pending_action — engine emits with `attemptId`; bridge preserves UUID v4 round-trip.
- C44 usage — engine emits; bridge maps v6 `LanguageModelUsage` (nested `inputTokenDetails`).
- C45 error — engine emits; bridge normalises Error envelope.
- C46 turn_complete — engine emits StopReason; bridge maps from v6 FinishReason (6-value enum).
- C47 canvas / C48 todo_update / C49 tool_progress / C50 pending_input / C51 harness_shape / C52 stream_state — all engine-emitted side-channel events; bridge routes via `data-{name}` UIMessageChunk parts.

**Intelligence caveats — v0.7a status:**
- **I1** engineMeta race risk → Phase 1 wires `streamText` through `bridgeAISDKStream`; bridge is stateless across instances, no race.
- **I2** handleAdviceResults timing → audric/web concern; v0.7a does not move it.
- **I3** ChainFact = `UserMemory source='chain'` → doc-fix only; ship with R7 cross-repo doc sweep at Phase 4.
- **I4** buildAdviceContext is `take:5` → doc-fix only; ship with R7.
- **I5** Resume route demotes Reasoning Engine (Haiku/low) → doc invariant; preserved verbatim.

Every other behavior is **audric-web** or **deferred** under v0.7a — i.e., the engine PR doesn't touch the implementation. It still must verify the wire shape that surfaces via the engine's public API hasn't drifted.
