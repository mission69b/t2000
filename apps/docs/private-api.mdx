The **Private & Confidential API** is an OpenAI-compatible inference endpoint at `api.t2000.ai`. Point any OpenAI SDK at it and call frontier + open models through **one key**, in two privacy tiers:

- **Private** (default, every model) — **zero data retention**: providers are contractually bound not to store or train on your prompts.
- **Confidential** (`phala/*` models) — inference runs in a **verified GPU-TEE** (hardware enclave), and every response carries a **signed receipt anchored on Sui** that you can **check yourself** with `t2 verify` — the only Sui-native *verifiable* confidential inference.

Metered per token from a single credit balance. Manage everything — keys, credit, usage, the model catalog — at the developer console: **[platform.t2000.ai](https://platform.t2000.ai)**. Sign in with Google (zkLogin); it's the same Passport identity + credit balance as [Audric](https://audric.ai), so one account spans the app and the API.

Your edge over a raw provider: **private by default, verifiably confidential when it matters, every model behind one key, and pay-as-you-go with no per-provider accounts.**

## Quick start

<CodeGroup>

```bash curl
curl https://api.t2000.ai/v1/chat/completions \
  -H "Authorization: Bearer $T2000_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "zai/glm-5.2",
    "messages": [{ "role": "user", "content": "Explain x402 in one sentence." }]
  }'
```

```python python
from openai import OpenAI

client = OpenAI(
    base_url="https://api.t2000.ai/v1",
    api_key="sk-...",  # from Settings → API keys
)

resp = client.chat.completions.create(
    model="zai/glm-5.2",
    messages=[{"role": "user", "content": "Explain x402 in one sentence."}],
)
print(resp.choices[0].message.content)
```

```typescript typescript
import OpenAI from "openai";

  baseURL: "https://api.t2000.ai/v1",

```

## Pay-per-call over x402 — no key, no account

Agents can pay **per call in USDC** with **no API key** — the t2000 Private API is a first-party [x402](/agent-payments) service on the gateway. Call it with the existing `t2 pay` / `t2000_pay` (the gateway handles `402` → pay → retry):

```bash
t2 pay https://x402.t2000.ai/t2000/v1/chat/completions \
  --data '{"model":"zai/glm-5.2","messages":[{"role":"user","content":"Explain x402 in one sentence."}]}' \
  --max-price 0.10
```

- **No key / no account** — pay from your wallet's USDC, gasless. The agent-native path (vs the key-based one above).
- **Models:** the **open + confidential** set (`zai/glm-5.2`, `deepseek/deepseek-v3.2`, `openai/gpt-oss-120b`, and the `phala/*` confidential models) — browse the live list at `GET /t2000/v1/models` (free). Frontier + large contexts use a **key** (above).
- **Capped** — flat per-call price with an output + context cap (what keeps no-key pricing predictable). For uncapped/per-token precision, use a key, or watch for the **max-hold settlement** tier (next).

## Streaming

Set `"stream": true` for server-sent `chat.completion.chunk` events (OpenAI-compatible), terminated by `data: [DONE]`. Pass `"stream_options": { "include_usage": true }` to receive a final token-usage chunk.

```typescript
const stream = await client.chat.completions.create({
  model: "zai/glm-5.2",
  messages: [{ role: "user", content: "Write a haiku about Sui." }],
  stream: true,
});
for await (const chunk of stream) {
  process.stdout.write(chunk.choices[0]?.delta?.content ?? "");
}
```

## Privacy tiers

Two rungs, selectable per model (the `privacy` field in `GET /v1/models`):

- **Private (default, every model)** — **zero data retention (ZDR)**: providers are contractually bound not to store or train on your prompts.
- **Confidential (`phala/*` models)** — inference runs inside a **GPU-TEE (hardware enclave, Intel TDX + NVIDIA CC)**, so prompts are encrypted in use, not just contractually protected. The gateway **verifies the enclave's attestation before forwarding** your prompt and **fails closed** if it can't (genuine-attested-enclave-or-refuse). Each response returns an **`x-receipt-id`** (non-streaming: HTTP header; streaming: `x_receipt_id` on the final usage chunk) — a signed per-response receipt.

<Note>
  **What "Confidential" proves — stated honestly.** You can verify that the compute ran in a genuine GPU-TEE and that the exact response is committed in a signed, Sui-anchored receipt. On its forwarding leg the gateway still terminates TLS and sees plaintext (zero data retention, retains nothing) — full end-to-end encryption where even t2000 never sees plaintext is the deferred next rung. We claim exactly what the verifier proves, no more.
</Note>

### Verify confidentiality

```bash
# Is a model backed by a genuine, freshly-attested GPU-TEE?
curl "https://api.t2000.ai/v1/aci/attestation?model=phala/glm-5.2"
# → { verified: true, enforced: true, teeType: "tdx", workloadId, tlsSpkiSha256, signingKey, … }

# Add ?nonce=<hex> to get the FULL attestation report bound to your nonce —
# the raw TDX quote + keyset + endorsement + provenance, so you can DCAP-verify
# it yourself and prove freshness (the quote's report_data commits your nonce):
curl "https://api.t2000.ai/v1/aci/attestation?model=phala/glm-5.2&nonce=$(openssl rand -hex 32)"
# → { …summary, nonce, report: { attestation: { evidence.quote, report_data, workload_keyset, keyset_endorsement, … } } }

# Fetch the signed receipt for a specific confidential response:
curl "https://api.t2000.ai/v1/aci/receipts/{x-receipt-id}"

# Resolve the receipt's attested-session (verified upstream channel, typed TCB
# claims, channel binding, evidence) — from upstream.verified.session_id:
curl "https://api.t2000.ai/v1/aci/sessions/{session-id}"

# Look up a receipt's on-chain Sui anchor (the tx digest):
curl "https://api.t2000.ai/v1/aci/anchor/{x-receipt-id}"
```

The receipt commits the request/response **hashes** (not bodies) to the attested workload and is signed by a `signingKey` from the attestation. `t2 chat --model phala/…` surfaces the receipt id inline (`🔒 confidential · attested · receipt …`).

### Anchor + verify (Sui-native)

A confidential receipt can be **anchored on Sui** — emitting a tamper-evident, publicly-timestamped `ReceiptAnchored` event committing the response `wire_hash` + attested `workload_id`. This is the wedge: the only Sui-native verifiable-inference anchor.

```bash
# Anchor a receipt on-chain (idempotent — returns the existing tx if already anchored):
curl -X POST "https://api.t2000.ai/v1/aci/anchor/{x-receipt-id}"
# → { anchored: true, txDigest, explorer }

# Then verify it yourself — fails closed on any mismatch:
t2 verify {x-receipt-id}
```

`t2 verify` does two **trustless** checks: it reads the on-chain `ReceiptAnchored` event **straight from a Sui fullnode** (t2000 can't forge it) and confirms the committed `wire_hash` + `workload_id` match the signed receipt; and it **recovers the receipt's secp256k1 signer** (recomputing the dstack ACI canonical bytes) and matches it to the **attested** receipt-signing key — so an altered or forged receipt fails. It also surfaces the upstream's **typed TCB claims** (`tee_attested`, `gpu_attested`, `tcb_up_to_date`, …, each with its source) and the attested-session id. It prints a per-check report and exits non-zero if anything doesn't line up. (Programmatic: `agent.verify(receiptId)` / `verifyReceipt(receiptId)` in `@t2000/sdk`; `t2000_verify` over MCP.)

<Note>
  The **Sui anchor** and **receipt signature** are verified trustlessly client-side today. The **TDX quote** is verified + enforced **server-side** (the receipt also embeds the per-request `upstream.verified` attestation evidence); local `dcap-qvl` quote re-verification is the one documented hardening step toward full no-trust. We state exactly what's proven — a wrong "verifiable" claim is worse than honest ZDR.
</Note>

## Pricing

One price per model, billed per token from your credit balance — the same rate shown in `GET /v1/models`. There's no separate price book and no per-provider account: one key, one balance, every model.

## Rate limits

Each key is capped at **120 requests/minute** (an abuse guard). Exceeding it returns `429`. Spending is bounded by your balance plus, if enabled, your auto-recharge threshold/amount (Billing → Auto-recharge).

## Errors

Errors use the standard OpenAI envelope, so existing SDKs surface them correctly:

```json
{ "error": { "message": "Invalid API key.", "type": "invalid_request_error", "code": "invalid_api_key", "param": null } }
```

| Status | When |
|---|---|
| `401` | Missing or invalid API key |
| `402` | Insufficient credit — add credit at [platform.t2000.ai](https://platform.t2000.ai) |
| `404` | Unknown model — see `GET /v1/models` |
| `429` | Rate limit exceeded (120 req/min per key) |
