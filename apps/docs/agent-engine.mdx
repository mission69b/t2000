`@t2000/engine` is the agent **harness library** for conversational finance on Sui — the brain behind [Audric](https://audric.ai). It implements **Audric Intelligence**, a 4-system stack: 26 financial tools, a 12-guard reasoning pipeline, a USD-aware permission resolver, 4-layer prompt assembly, MCP client/adapters, and the host-composition primitives a host wires into Vercel AI SDK's own agent loop.

> _Not a chatbot. A financial agent._ Every action still waits on Audric Passport's tap-to-confirm. The engine never moves money on its own.

<Note>
  **[S.391 — 2026-06-09] The runnable `AISDKEngine` loop was retired.** The engine no longer ships a runnable agent (the `AISDKEngine.submitMessage()` async-generator loop + its `EngineEvent` SSE/checkpoint/event-bridge transport had zero live consumers). The host owns the AI SDK loop (`Experimental_Agent`) and composes the engine's primitives — tools, guards, permissions, prompt assembly, `buildToolContext` / `buildInternalContext` / `buildStepFinishHandler` — into it. This is `@t2000/engine` **5.0.0** (MAJOR). The snippets below reflect that model.
</Note>

## Install

```bash
npm install @t2000/engine @t2000/sdk
```

Requires Node.js 18+ · TypeScript 5+ recommended.

## Quick Start (host composition)

The host owns the AI SDK loop and wires the engine's primitives into it: build the tool set, build the per-turn `ToolContext` + `InternalContext`, and run guard state + post-write refresh via the step-finish handler.

```typescript
import { Experimental_Agent as Agent } from 'ai';
import { gateway } from '@ai-sdk/gateway';
import {
  READ_TOOL_SET,
  WRITE_TOOL_SET,
  buildToolContext,
  buildInternalContext,
  buildStepFinishHandler,
  DEFAULT_GUARD_CONFIG,
} from '@t2000/engine';
import { T2000 } from '@t2000/sdk';

const agent = await T2000.create();

const config = { agent, anthropicApiKey: process.env.ANTHROPIC_API_KEY };
const toolContext = buildToolContext(config, { signal: AbortSignal.timeout(60_000) });
const messages: { role: string; content: unknown }[] = [];
const internalContext = buildInternalContext({
  toolContext,
  walletAddress: agent.address,
  guards: DEFAULT_GUARD_CONFIG,
  getMessages: () => messages,
});

const llm = new Agent({
  model: gateway('anthropic/claude-sonnet-4-5'),
  tools: { ...READ_TOOL_SET, ...WRITE_TOOL_SET },
  experimental_context: internalContext,
  onStepFinish: buildStepFinishHandler(internalContext, { sessionSpendUsd: 0 }),
});

for await (const part of llm.stream({ prompt: 'What is my balance?' }).fullStream) {
  if (part.type === 'text-delta') process.stdout.write(part.text);
}
```

Write tools yield through AI SDK's native `needsApproval` round-trip (every write taps to confirm under zkLogin). `getDefaultTools()` is still exported if you want the combined read+write set as a single map.

## The 4 systems

| System | One-line | Lives in |
|---|---|---|
| 🎛️ **Agent Harness** | 26 tools (18 read + 8 write), one agent. | `tools/*`, `v2/tool-policy.ts`, `READ_TOOL_SET` / `WRITE_TOOL_SET` |
| ⚡ **Reasoning Engine** | Thinks before it acts. 12 guards, 3 priority tiers. | `classify-effort.ts`, `guards.ts`, `v2/step-finish.ts` |
| 🧠 **Memory (MemWal)** | Knows your finances + remembers your patterns. | Engine: `prepareStep` + `MemoryStore` interface. Backend: `@mysten-incubation/memwal` (audric-side) |
| 📓 **AdviceLog** | Remembers what it told you. | `record_advice` audric tool + `buildAdviceContext()` |

The engine package owns **Agent Harness** and **Reasoning Engine** in code, plus the `MemoryStore` injection point for **Memory**. The MemWal vector backend and the AdviceLog Prisma model are audric-side. The agent orients itself with fresh read-tool calls (`balance_check`, `savings_info`, `rates_info`) each turn — no pre-injected financial snapshot.

## Agent Harness — 26 tools

| Type | Count | Examples |
|---|---|---|
| **Read** | 18 | `balance_check`, `savings_info`, `health_check`, `rates_info`, `transaction_history`, `swap_quote`, `portfolio_analysis`, `token_prices`, `spending_analytics`, `yield_summary`, `resolve_suins`, `pending_rewards`, `render_canvas`, … |
| **Write** | 8 | `save_deposit`, `withdraw`, `send_transfer`, `borrow`, `repay_debt`, `claim_rewards`, `harvest_rewards`, `swap_execute` |

Read tools dispatch in parallel mid-stream (AI SDK v6 `tool-call` event). Write tools require user confirmation — the engine yields a `pending_action` event; the host client signs + executes; the engine resumes with the result.

**Saveable assets:** USDC + USDsui (per `OPERATION_ASSETS.save` / `borrow`). Everything else is sendable / swappable only.

## Reasoning Engine — 12 guards

Every write goes through 12 guards across 3 priority tiers. Guards return one of:

- `{ pass: true }` — silent, allowed
- `{ pass: true, hint: '…' }` — allowed, hint injected for LLM context
- `{ pass: true, warning: '…' }` — allowed, warning surfaced in confirm card
- `{ pass: false, reason: '…' }` — **BLOCKED**, agent narrates and re-asks

| Tier | Guards |
|---|---|
| **Safety** | Health Factor, Balance Validation, Address Source / Address Scope, Irreversibility |
| **Financial** | Large Transfer (USD threshold), Slippage, Asset Intent |
| **UX** | Swap Preview, Retry Protection, Stale Data (post-execution hint) |

Plus adaptive thinking effort per turn (cheap vs. expensive reasoning), preflight input validation (no LLM round-trip if the input is malformed), and prompt caching for the system prompt + tool definitions.

## Memory — MemWal recall

Engine assembles the system prompt in 4 layers via `prepareStep`:

1. Base `systemPrompt`
2. `<memory_recall>` — top-K vector facts from MemWal
3. `skillRecipeBlock` (multi-step orchestration playbooks)
4. `messages[]` (conversation history)

Tests use `InMemoryMemoryStore` (default). Production Audric injects a MemWal-backed store via `@mysten-incubation/memwal`. The host owns `prepareStep` assembly; the engine provides the `MemoryStore` interface + the in-memory default.

```typescript
import { InMemoryMemoryStore } from '@t2000/engine';

const memory = new InMemoryMemoryStore();
// In the host's prepareStep: recall top-K facts and prepend a
// <memory_recall> block to the system prompt before the skill recipe.
const facts = await memory.recall(latestUserMessage);
```

Wire your own backend by implementing the `MemoryStore` interface (`recall(query)` returns top-K facts; `remember(text)` ingests a fact post-turn).

## AdviceLog — never contradict yourself

Every recommendation the agent makes is logged via the `record_advice` tool (audric-side). The last 30 days of advice are hydrated into the system prompt each turn so the chat doesn't contradict itself across sessions ("I told you to deposit USDC last week, here's the update").

## Stream events

The host consumes AI SDK v6's native `fullStream` parts (`text-delta`, `reasoning-delta`, `tool-call`, `tool-result`, `finish`, …) — the engine no longer defines its own `EngineEvent` union (removed in 5.0.0 with the runnable loop). Write tools surface through AI SDK's `needsApproval` round-trip; the engine still owns the **`PendingAction`** shape the host renders on the confirm card:

```typescript
import type { PendingAction } from '@t2000/engine';

// PendingAction carries the confirm-card payload: description, modifiableFields,
// cetusRoute / steps[] for bundles, borrowApyBps, currentHF / projectedHF,
// quoteAge / canRegenerate, plus the attemptId (UUID v4) the host round-trips
// through the user-confirm event to correlate the resume.
```

## Tool permission levels

| Level | Behavior |
|---|---|
| `auto` | Read tools (`balance_check`, `swap_quote`, …). Execute without approval. |
| `confirm` | Write tools (`save_deposit`, `swap_execute`, …). Yield `pending_action` for client-side signing. |
| `explicit` | Manual-only, never dispatched by LLM. |

USD-aware resolver (`resolvePermissionTier(operation, amountUsd, config)`) downgrades writes dynamically based on USD value. Under zkLogin (Audric web-v2), every write taps to confirm — the auto-execute path is structurally disabled by design. Under server-signing hosts (`@t2000/cli` engine usage with a key), sub-threshold writes can auto-execute per the configured preset (`conservative` / `balanced` / `aggressive`).

## Sessions

```typescript
import { MemorySessionStore } from '@t2000/engine';
```

- **`MemorySessionStore`** — default `SessionStore` (in-memory). Hosts inject Redis / Postgres backends.

Streaming is the host's concern: it streams AI SDK `fullStream` parts directly. The engine's `EngineEvent` SSE helpers (`serializeSSE` / `parseSSE`), the `StreamCheckpointStore`, and `budgetToolResult` were removed in 5.0.0 — Audric does live-stream resume with the Redis-backed [`resumable-stream`](https://www.npmjs.com/package/resumable-stream) package over AI SDK `UIMessage` chunks, and tool-result budgeting is configured per-tool via `maxResultSizeChars`.

## MCP integration (external MCPs)

```typescript
import { McpClientManager, NAVI_MCP_CONFIG, McpPromptAdapter } from '@t2000/engine';

const mcp = new McpClientManager();
await mcp.connect(NAVI_MCP_CONFIG);                // NAVI MCP for live lending reads
const tools = await mcp.listTools();               // merge into the Agent's tools map
```

`McpClientManager` is backed by `@ai-sdk/mcp`. `McpPromptAdapter` is an extension point for consuming MCP prompts as skill recipes — currently dormant by design (it's the "speak any Sui protocol's MCP" seam, kept unwired until a downstream consumer needs it).

## Exposing tools over MCP

The engine's tools are plain AI SDK `tool()` instances (`getDefaultTools()`), so any MCP server framework can wrap them directly — there is no engine-side MCP server adapter (the old `buildMcpTools` / `registerEngineTools` helpers were removed in v3.0.0). For the supported t2000 wallet MCP surface, use the standalone [`@t2000/mcp`](/agent-wallet) package (9 wallet + payment tools backed by `@t2000/sdk`).

## What's in scope vs. host

**Engine owns (the library surface the host composes):**

- Tool registry, dispatch policy, permission gates (`READ_TOOL_SET` / `WRITE_TOOL_SET`, `v2/tool-policy.ts`)
- 12 guards + USD-aware permission resolver (run via `buildStepFinishHandler` + `runGuardsForTool`)
- Per-turn context builders (`buildToolContext`, `buildInternalContext`)
- Prompt-assembly primitives + `MemoryStore` interface (host assembles the 4 layers in its `prepareStep`)
- MCP client + prompt adapters

**Host owns (e.g. Audric `apps/web-v2`):**

- The AI SDK loop itself (`Experimental_Agent`), LLM provider keys + model selection
- `prepareStep` (4-layer system prompt assembly) + the `MemoryStore` backend (MemWal in production, in-memory in tests)
- `SessionStore` backend + live-stream resume (Redis `resumable-stream`)
- Transaction signing + tap-to-confirm UI (write tools yield via `needsApproval`, host signs)
- `record_advice` tool implementation (AdviceLog Prisma model is audric-side)

## Architecture

```
User message
    │
    ▼
Host: Experimental_Agent (AI SDK v6) — owns the loop
    │
    ├── model + prepareStep (host assembles 4-layer system prompt)
    │       └── <memory_recall> ← engine MemoryStore.recall(latestUserMessage)
    │
    ├── tools: { ...READ_TOOL_SET, ...WRITE_TOOL_SET }   ← engine
    │       ├── 18 read tools (isConcurrencySafe → parallel mid-stream)
    │       └── 8 write tools (needsApproval round-trip → host signs)
    │
    ├── experimental_context: buildInternalContext(...)  ← engine
    │       └── buildToolContext(config) per turn (BlockVision/NAVI/Sui + caches)
    │
    └── onStepFinish: buildStepFinishHandler(...)        ← engine
            ├── runGuardsForTool (12 guards, 3 priority tiers)
            └── post-write portfolio refresh + sessionSpendUsd
```
